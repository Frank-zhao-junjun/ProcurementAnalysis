# AI 助手集成设计修订清单

**来源文档**: `2026-04-11-ai-assistant-integration-design.md`
**修订目标**: 将五方评审意见转化为可执行 action items，作为下一版设计评审前的必改清单。
**适用范围**: 采购驾驶舱 AI 助手 Phase 1
**状态**: 待修订

---

## 1. 修订结论

当前方案不建议直接进入开发。需要先完成以下四类修订：

1. **上下文与 Prompt 重构**：从“全量系统消息注入”改为“最小上下文 + 按需取数”。
2. **代理架构补强**：补齐超时降级、缓存、限流、熔断、可观测性设计。
3. **业务口径落地**：把降本、TCO、物流、返利、预警责任等规则写成强约束文档。
4. **安全合规收口**：明确日志脱敏、留存周期、访问权限、第三方数据边界。

---

## 2. 必改项（P0）

### P0-1. 重构上下文注入策略

**问题**

当前方案将 `current_kpis`、`p0_alerts` 等整块数据直接拼入系统消息，存在以下风险：

- 用户问题权重被稀释
- 实时数据与知识库示例值冲突
- 数据外发范围过大
- Token 消耗不可控

**修订动作**

- 将系统消息改为仅保留角色、回答规则、边界约束。
- 将页面状态拆分为三层：
  - `session context`: 用户角色、当前模块、当前时间范围
  - `ui context`: 当前 tab、筛选器、选中对象、可见图表
  - `data context`: 仅按问题所需动态获取
- 增加 `get_dashboard_data` 取数插件或后端工具接口，按需返回 KPI、预警、图表明细。
- 不再把整页 KPI/预警 JSON 全量塞入 system message。

**验收标准**

- 设计文档中出现明确的数据分层定义。
- 系统消息示例不包含整段 `JSON.stringify(context.current_kpis)` 之类的全量注入。
- 文档中新增按需取数时序说明。

**建议责任人**

- AI/Bot 专家
- 后端架构专家

### P0-2. 补齐页面状态实时同步设计

**问题**

当前上下文只在发送消息时采集，无法覆盖 Tab 切换、筛选变更、图表点击、高亮变化等实时状态。

**修订动作**

- 在设计中新增 `state sync` 机制。
- 明确以下事件触发上下文更新：
  - Tab 切换
  - 维度切换
  - 图表点击
  - KPI 卡片选中
  - 筛选器变化
- 增加上下文对象 schema，至少包含：
  - `active_tab`
  - `active_module`
  - `selected_dimension`
  - `active_filters`
  - `selected_entity`
  - `highlighted_chart`

**验收标准**

- 文档中新增状态同步章节或序列图。
- 明确哪些 UI 事件会触发上下文刷新。
- 设计中不再只有“发送消息前采集一次”这一条链路。

**建议责任人**

- 前端技术专家

### P0-3. 明确超时降级与高可用策略

**问题**

当前代理层仅定义 `timeout: 10000`，没有说明超时后的降级路径、缓存命中策略、并发高峰保护，也没有高可用部署说明。

**修订动作**

- 增加代理层高可用拓扑：至少说明多实例 + 负载均衡。
- 增加以下运行策略：
  - 单用户限流
  - 全局并发保护
  - 高频问题缓存
  - 熔断与恢复窗口
  - 超时后的前端提示与降级回答
- 评估是否改为流式响应，而非固定 `stream: false`。

**验收标准**

- 文档中新增“超时/失败/限流/熔断”处理表。
- 明确回答“Coze API 超过 10 秒怎么办”。
- 明确哪些问题允许缓存、缓存多久、缓存键如何定义。

**建议责任人**

- 后端/架构专家

### P0-4. 补充业务口径主文档

**问题**

当前知识库结构方向正确，但缺少足以支撑真实追问的强口径定义。典型风险包括：

- 降本是否含物流/返利/税费
- 上期单价如何定义
- TCO 的费用项范围
- 预警是否包含责任人、处理状态、到货计划

**修订动作**

- 在知识库中新增或强化以下文档：
  - `cost-calculation.md`
  - `tco-boundary.md`
  - `price-benchmark-rule.md`
  - `alert-disposition-rule.md`
- 每个业务口径文档必须包含：
  - 定义
  - 纳入项
  - 排除项
  - 示例问答
  - 常见误解

**验收标准**

- 文档明确回答“本月降本是否包含物流费用”。
- 至少 10 个高频追问有口径答案。
- AI 示例回答引用业务口径而非泛化表述。

**建议责任人**

- 采购业务专家
- AI/Bot 专家

### P0-5. 收紧日志与数据外发边界

**问题**

当前方案记录原始问题文本，并将较完整的业务上下文发送给第三方平台，存在合规与敏感数据泄露风险。

**修订动作**

- 明确日志默认不记录原始消息全文。
- 增加日志脱敏规则：
  - 供应商名称
  - 金额
  - 联系方式
  - 内部编码
- 明确日志保留周期、访问审批、审计方式。
- 明确 Coze 平台对输入数据的使用边界和是否参与模型训练。
- 增加“最小必要传输”原则描述。

**验收标准**

- 文档新增日志治理小节。
- 文档新增第三方数据处理边界说明。
- 不再默认记录完整问题与完整上下文。

**建议责任人**

- 信息安全专家
- 后端/架构专家

## 3. 高优先级修订项（P1）

### P1-1. 细化前端生命周期与内存管理

**修订动作**

- 说明 AI 组件初始化、销毁、重建时机。
- 说明与 ECharts 交互时的事件解绑和高亮恢复策略。
- 增加异常场景：页面长期驻留、频繁切换 Tab、组件收起/展开、浏览器缩放。

**验收标准**

- 文档中出现生命周期清单。
- 明确不会因 AI 组件导致 ECharts 实例泄漏。

### P1-2. 重新设计快捷按钮与 Top 50 高频问题

**修订动作**

- 快捷按钮从演示型问题改成真实晨会型问题。
- Top 50 FAQ 按场景分组，而不是只按主题分组。

**建议优先补充场景**

- 预算偏差归因
- 缺料与到货承诺
- 二供替代与询比价
- 降本构成拆解
- 预警责任人与处置进度

**验收标准**

- 快捷按钮至少覆盖“结论、归因、预警、行动”四类问题。
- FAQ 中包含追问型问题，而不仅是单轮事实问答。

### P1-3. 增加可观测性设计

**修订动作**

- 为每次请求定义 `trace_id`。
- 增加核心监控指标：
  - 请求量
  - 成功率
  - 95/99 响应时间
  - 缓存命中率
  - 降级触发次数
  - 用户反馈有用率

**验收标准**

- 文档中有监控指标表。
- 错误排查可以串联前端、代理、Coze 三段链路。

### P1-4. 增加安全传输与来源控制设计

**修订动作**

- 明确 HTTPS 强制、HSTS、CORS 白名单。
- 补充 JWT 的过期、刷新、存储方式建议。
- 明确是否允许 iframe 嵌入、是否需要 CSP。

**验收标准**

- 安全章节中有传输层与前端令牌存储规则。

## 4. 体验与可维护性修订项（P2）

### P2-1. 补充响应式与大屏遮挡说明

**修订动作**

- 说明固定 1920 宽度场景下 AI 组件的默认停靠位。
- 补充小屏/缩放/投屏时的遮挡处理逻辑。
- 增加“最小化仅保留入口按钮”方案。

### P2-2. 增加示例对话集

**修订动作**

- 每个模块至少补 3 组示例：
  - 事实查询
  - 趋势解读
  - 追问口径/行动建议

### P2-3. 增加版本治理机制

**修订动作**

- 为知识库文档与 Prompt 设计版本号。
- 明确上线前评审人和变更审批流程。

## 6. 进入开发前的通过条件

只有同时满足以下条件，才建议从设计阶段进入开发阶段：

1. P0 项全部关闭。
2. 至少 1 名采购业务使用方确认关键口径文档可用。
3. 安全专家确认日志与第三方数据边界可接受。
4. 后端专家确认超时降级和缓存策略可实现。
5. 前端专家确认状态同步与生命周期方案可落地。

---

## 7. 推荐下一步

建议按以下顺序推进修订：

1. 先改上下文注入、代理降级、安全边界三个 P0。
2. 再补业务口径和 FAQ 场景。
3. 最后统一刷新 UI 细节、快捷按钮和示例对话。
