# AI 助手 v2 同步评审与开发前门禁检查

**评审对象**: `2026-04-11-ai-assistant-integration-design-v2.md`  
**评审日期**: 2026-04-11  
**形式**: 虚拟同步评审  
**结论类型**: 设计门禁检查

---

## 1. 评审目标

本次同步评审用于确认两件事：

1. v2 设计是否已关闭上一轮评审识别的 P0 风险。
2. 是否满足 v2 文档第 12.2 节定义的开发前门禁条件。

---

## 2. 参会专家与结论

| 专家角色 | 关注范围 | 结论 |
| -------- | -------- | ---- |
| 前端技术专家 | 状态同步、生命周期、性能、大屏适配 | 通过 |
| 后端/架构专家 | 高可用、降级、缓存、限流、熔断、可观测性 | 通过 |
| AI/Bot 专家 | Prompt 最小化、按需取数、幻觉控制、口径漂移 | 通过 |
| 采购业务专家 | 业务口径、FAQ 场景、快捷提问、追问兜底 | 通过 |
| 信息安全专家 | 日志脱敏、最小必要传输、认证、第三方数据边界 | 有条件通过 |

---

## 3. 同步评审纪要

### 3.1 前端技术专家意见

结论: 通过

#### 前端已关闭项

- v1 的“发送前一次性抓取上下文”已替换为 `dashboardStateBus` 实时状态同步。
- AI 组件生命周期、单例模式、事件解绑、图表高亮前校验等要求已明确。
- 1920 大屏停靠位和遮挡时自动收起策略已定义。

#### 前端评审判断

前端侧 P0 风险已完成设计闭环，可进入开发。

### 3.2 后端/架构专家意见

结论: 通过

#### 后端已关闭项

- 代理层职责从“API 转发器”升级为“高可用控制层”。
- 已定义多实例、负载均衡、Redis、健康检查。
- 已定义 5 秒提示、10 秒降级、连续失败熔断。
- 已补齐缓存、限流、熔断、trace_id 和监控埋点设计。

#### 后端评审判断

架构层阻塞问题已关闭，可进入开发。

### 3.3 AI/Bot 专家意见

结论: 通过

#### AI 已关闭项

- system prompt 已收敛为长期稳定规则，不再承载全量业务数据。
- 已改为“按需工具取数”，避免用户问题权重被大块上下文淹没。
- 幻觉控制、口径缺失兜底、来源引用规则已明确。

#### AI 评审判断

AI 方案的核心 P0 风险已关闭，可进入开发。

### 3.4 采购业务专家意见

结论: 通过

#### 业务已关闭项

- 降本、TCO、物流、返利、价格偏差、预警责任等口径已升级为独立知识层。
- 快捷提问已从演示型按钮改为晨会型问题。
- FAQ 分组方式已向真实业务场景对齐。
- 对口径未定义的情形已给出明确兜底策略。

#### 业务评审判断

业务侧允许进入开发，但要求在开发阶段优先落地关键口径文档。

### 3.5 信息安全专家意见

结论: 有条件通过

#### 安全已关闭项

- 已明确最小必要传输，不再默认发送整页状态。
- 已明确 JWT、CORS 白名单、HTTPS、HSTS 基线。
- 已明确日志默认不记录原始问题全文和完整 KPI / 预警快照。

#### 安全未关闭项

- Coze 平台对 API 请求/响应是否持久化存储、存储多久，仍未形成确定性结论。

#### 安全评审判断

安全设计总体方向可接受，但在进入开发前仍需补齐第三方平台持久化存储边界说明。

---

## 4. P0 问题关闭情况总览

| P0 项 | 来源 | 当前状态 | 说明 |
| ---- | ---- | -------- | ---- |
| 上下文注入重构 | v1 评审 | 已关闭 | 已改为最小 prompt + 按需取数 |
| 页面状态实时同步 | v1 评审 | 已关闭 | 已引入 `dashboardStateBus` |
| 超时降级与高可用 | v1 评审 | 已关闭 | 已定义缓存、限流、熔断、降级 |
| 业务口径主文档 | v1 评审 | 已关闭 | 已列为独立知识层和强约束 |
| 日志与第三方数据边界 | v1 评审 | 部分关闭 | 日志与隐私提示已关闭，平台持久化存储边界未确认 |

**综合判断**: 4 项已关闭，1 项部分关闭。

---

## 5. 开发前门禁检查（对应第 12.2 节）

### 5.1 门禁项检查表

| 门禁条件 | 要求 | 当前结果 | 判定 |
| -------- | ---- | -------- | ---- |
| 1 | P0 修订项全部关闭 | 仍有平台持久化存储边界未确认 | 未通过 |
| 2 | 至少 1 名采购业务使用方确认关键口径 | 已由采购业务专家确认 | 通过 |
| 3 | 安全专家确认日志与第三方边界可接受 | 当前为“有条件通过” | 未通过 |
| 4 | 后端专家确认缓存与降级方案 | 已确认 | 通过 |
| 5 | 前端专家确认状态同步与生命周期方案 | 已确认 | 通过 |

### 5.2 门禁结论

### 当前门禁结论: 允许带条件进入开发阶段

说明如下：

- 设计级 P0 风险已基本关闭。
- 安全侧剩余问题已收敛为单一外部确认项：第三方平台持久化存储边界。
- 项目组决定接受该风险，并限定开发范围为“测试数据/脱敏数据 + 非生产放量”。
- 在真实生产数据接入和试运行前，必须再次执行安全复核。
- 在试运行或正式上线前，必须完成 `2026-04-11-ai-assistant-production-readiness-checklist.md` 全项核对。

---

## 6. 阻塞项清单

以下事项不再作为“开发启动阻塞项”，但保留为“生产放行前阻塞项”：

1. 获取并记录 Coze 平台关于 API 请求/响应是否持久化存储的结论。
2. 获取并记录如存在持久化存储，其存储周期和删除机制。

---

## 7. 决议

### 7.1 本轮会议决议

- 同意认可 v2 文档已关闭大部分设计级 P0 风险。
- 同意项目以“带条件放行”方式进入开发。
- 要求将第三方平台持久化存储边界列为生产放行前必清事项。

### 7.2 建议后续动作

建议按以下顺序推进：

1. 立即进入开发计划拆解和非生产实现。
2. 并行补齐 Coze 平台持久化存储边界说明。
3. 在接入真实生产数据前，由安全专家进行一次快速复核。
4. 在试运行或正式上线前，重新执行门禁检查与 production readiness checklist。

---

## 8. 最终结论

同步评审结果: v2 设计方向正确，技术与业务主体已达成一致。  
开发门禁结果: 带条件放行。  
生产放行前提: Coze 平台持久化存储边界确认后，需重新提交门禁并完成 production readiness checklist。
