## 1. 架构决策记录

### 1.1 为什么选择单文件而非框架？

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| **单文件HTML (当前)** | 零构建、直接部署、CDN引入ECharts | 文件长(2119行)、无组件化 | ✅ |
| Next.js | 组件化、SSR、生态丰富 | 需要构建步骤、增加复杂度 | ❌ |
| Vue/React SPA | 组件化、状态管理 | 需要npm/build工具链 | ❌ |

**决策理由**:
- 用户明确要求"无构建步骤"、"可直接在浏览器打开"
- 大屏场景通常部署在内网/专用设备，不需要SEO
- ECharts CDN引入足够稳定，无需打包优化
- 未来如需扩展，可迁移到Vite/Next.js

### 1.2 为什么一次性初始化全部37个图表？

| 策略 | 优点 | 缺点 |
|------|------|------|
| **全部初始化(当前)** | Tab切换即时展示，无闪烁 | 初始加载稍慢(~2s) |
| 懒加载(按Tab) | 初始加载快 | 切换时需等待渲染，体验差 |

**决策理由**: 
- 采购总监晨会场景下，切换等待是不可接受的
- 37个图表初始化总耗时约1.5~2s（ECharts原生性能优秀）
- 使用 `requestAnimationFrame` 延迟resize确保不阻塞UI

### 1.3 为什么维度切换用"分发器模式"而非"配置驱动"？

```javascript
// 分发器模式(当前): switch-case
function applyDimension(chartId, dimName) {
  switch(chartId) {
    case 'chartMonthlySavings': /* 特定处理 */ break;
    case 'chartBuSavings': /* 特定处理 */ break;
  }
}

// 配置驱动(备选):
const DIM_CONFIG = {
  'chartMonthlySavings': { seriesPath: 'series[0].data', dataKey: 'actual' }
}
```

**选择分发器的原因**:
- 每种图表类型的option结构差异巨大（柱状 vs Treemap vs 雷达）
- 配置驱动需要写一个通用的deep merge逻辑，复杂度更高
- 分发器虽然代码量稍多，但每个case清晰独立，易维护
- 14个chart的switch-case完全可控

## 3. CSS 设计笔记

### 3.1 响应式断点选择

```css
/* 默认: 1400px+ */
/* @media max-width: 1400px */ → 缩小KPI卡片间距
/* @media max-width: 1000px */ → 单列布局, 进一步压缩
```

为什么不用 `1920px` 作为断点？
- 用户反馈Chrome 100%缩放下溢出
- 实际使用场景包括笔记本(1366px)、普通显示器(1440px)
- `auto-fit/auto-fill` + `minmax()` 自适应比固定断点更灵活

### 3.2 颜色体系

采用"语义化颜色"而非"品牌色"：
- Cyan = 正向/主数据（用户直觉：蓝色=信息）
- Gold = 警示/目标（黄色=注意）
- Green = 达标/安全
- Red = 风险/危险
- Purple = 辅助维度（子公司/库存）

好处：用户看到颜色即知含义，无需图例对照。

## 4. AI助手设计笔记

### 4.1 为什么先用规则引擎而非LLM？

| 因素 | 规则引擎(当前) | LLM API(未来) |
|------|---------------|--------------|
| 延迟 | < 1s (本地匹配) | 2~5s (网络+生成) |
| 成本 | 免费 | 按token计费 |
| 准确性 | 100%可控 | 可能幻觉 |
| 扩展性 | 差(需手写规则) | 强(自然语言理解) |

**策略**: 先用规则引擎验证交互流程和UI，Phase 2 对接LLM。

### 4.2 typing动画时长

```
基础延迟: 800ms + random * 700ms = 800~1500ms
```
- 太短(<500ms)：用户感觉不到"思考"
- 太长(>2000ms)：用户以为卡了
- 800~1500ms：模拟人类阅读+组织语言的时间

## 5. 下钻弹窗设计笔记

### 5.1 为什么预埋而不做全量内容？

- 当前阶段重点是**架构搭建**和**交互通路验证**
- 每个KPI/图表的下钻内容差异大，需逐个设计
- 预埋=搭好骨架+通路，后续填充内容即可

### 5.2 Modal的3个Tab设计意图

| Tab | 内容规划 | Phase 1 | Phase 2 |
|-----|---------|---------|---------|
| 明细数据 | 表格形式展示原始数据 | placeholder | 真实数据表 |
| 趋势分析 | 子图表(更细粒度) | placeholder | 子ECharts实例 |
